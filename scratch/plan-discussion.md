Can the “lowest layer” be Rust even if you target LLVM?

Yes. A common pattern is:

    Axis compiler front-end in Rust (parser → AST → IR)

    Lower to LLVM IR (still from Rust)

    Use LLVM’s toolchain to emit object code / executable (or JIT)

Rust is “close to C” in the ways that matter here (predictable layout when you want it, explicit ownership, no GC), and you can keep your backend tight and deterministic.
The practical architecture
Option A: Rust front-end → LLVM IR text

Generate .ll (LLVM IR as text) and call llc / clang to produce binaries.

Pros: simplest, debuggable (you can open the .ll), no deep LLVM API binding pain
Cons: slower pipeline, you shell out
Option B: Rust front-end → LLVM via bindings (Inkwell / llvm-sys)

Build the LLVM module in-process and emit object code.

Pros: cleaner “real compiler” feel, can JIT, easier to integrate later
Cons: bindings/version friction, more moving parts

If you’re still proving the concept, A is often the fastest “credible” path.
What about runtime / “no Rust underneath”?

This is the key design point:

If Axis lowers to LLVM, then the runtime must be:

    either purely in LLVM-friendly code (C-ish ABI, no Rust stdlib assumptions), or

    a tiny runtime library you link in (written in Rust or C), but compiled down to native code too.

So yes, you can write runtime helpers in Rust — just treat them like C ABI functions:

    #[no_mangle]

    extern "C"

    avoid Rust-only calling conventions

    keep types ABI-safe (i64, u64, pointers)

That way, from LLVM’s point of view, it’s just calling C functions.
The “lowest layer” split that works well
Layer 0: Pure LLVM-lowerable core

    integers, bools, unit

    function calls

    if/let

    deterministic evaluation

This can compile to LLVM with no runtime beyond main.
Layer 1: Minimal runtime (Rust ok)

    printing

    timing / benchmarking hooks

    memory allocation if you later add strings/arrays

    maybe bigint or float ops later

Write these as extern "C" functions compiled into a static lib.
The crucial rule

Don’t let “Rust is close to C” tempt you into relying on Rust semantics.
Rely on ABI + LLVM, not on “Rust-ness”.

So: Rust as an implementation language = great.
Rust as the semantic substrate = not what you want if LLVM is the true backend.
If you want a super-clean path

A very sane staged approach:
